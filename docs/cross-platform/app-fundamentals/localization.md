# <a name="localization"></a>지역화

이 가이드에서는 개념을 소개 *국제화* 하 고 *지역화* 및 해당 개념을 사용 하 여 Xamarin 모바일 응용 프로그램을 생성 하는 방법에 대 한 지침에 대 한 링크입니다.

Xamarin 앱을 지역화 하는 기술 세부 정보로 직접 건너 뛰 려는 경우 다음 플랫폼 관련 방법 도움말 문서 중 하나를 사용 하 여 시작 합니다.

- [**Xamarin.Forms** ](~/xamarin-forms/app-fundamentals/localization/index.md) RESX 파일을 사용 하 여 플랫폼 간 지역화 합니다.
- [**Xamarin.iOS** ](~/ios/app-fundamentals/localization/index.md) 네이티브 플랫폼 지역화 합니다.
- [**Xamarin.Android** ](~/android/app-fundamentals/localization.md) 네이티브 플랫폼 지역화 합니다.

## <a name="i18n-and-l10n"></a>i18n 및 L10n

*국제화* 의 다른 언어를 표시 하 고 서로 다른 로캘 (예: 숫자 및 날짜 서식 지정)에 대 한 표시를 조정 가능한 코드를 작성 하는 프로세스입니다. 이 라고도 *세계화*합니다.

*지역화* 따릅니다 – 각 언어에 대 한 리소스 (예: 문자열 및 이미지)를 만들고 internationalize 앱 번들 화 하는 단계입니다.

국제화 i18n-18 문자 "i"와 "n" 사이의 축약형 종종 줄어듭니다. 지역화는 L10n에 – 10 개의 문자 "L"와 "n" 사이 마찬가지로 단축 됩니다.

## <a name="overview"></a>개요

이 문서에서는 국제화 및 지역화, 적용 되는 방식을 모바일 응용 프로그램 개발에 일반적으로 관련 된 개념을 소개 합니다.
이전에 하드 코드 된 디자인 하 고 있는 응용 프로그램을 작업을 빌드할 때 물어야 하지만 지역화에 대해 매개 변수화 해야 하는:

-   화면 레이아웃 및 텍스트
-   아이콘, 그래픽 및 색
-   비디오 및 사운드 파일
-   동적 텍스트와 텍스트 서식 (예: 숫자, 통화 및 날짜)
 - 오른쪽에서 왼쪽 (RTL) 언어에 대 한 레이아웃 변경 및
-   데이터 정렬입니다.

에 관계 없이 고품질의 지역화 된 응용 프로그램을 빌드할 수 앱 이러한 팁이 대상으로 하는 모바일 플랫폼 도와줍니다.


## <a name="design-considerations"></a>디자인 고려 사항

해당 콘텐츠를 지역화할 수 있도록 응용 프로그램을 설계 국제화를 라고 합니다. 언어 및 로캘 (이미지, 사운드 및 비디오 포함)에 따라 런타임에 로드할 문자열 다른 언어에 대 한 있도록 – 잘 설계 된 앱을 변경 하는 모든 리소스에 대해 허용 해야 하는 보다 국제화를 제대로 수행 및 조정할 수 있습니다. 서식 및 레이아웃에 대처할 수 있는 다양 한 크기 문자열.

이 섹션에서는 국제화 된 응용 프로그램을 빌드할 때 고려할 몇 가지 디자인 고려 사항을 설명 합니다.

### <a name="layouts-and-string-length"></a>레이아웃 및 문자열 길이

중국어 및 일본어로 문자열 매우 짧을 수 – 경우에 따라 하나 또는 두 개의 문자는 입력된 필드 레이블에 대 한 충분 한 의미를 가질 수 있습니다.

독일어 문자열 예를 들어 매우 길어질 수 있습니다. 경우에 따라 영어 단어는 비교적 짧은 하거나 잘린 또는 다른 레이아웃을 예기치 않게 리플로우 되기 – 다른 언어로 매우 긴 됩니다.

영어, 독일어 및 일본어로 iOS 홈 화면에서 몇 가지 항목에 대 한 문자열 길이 비교 합니다.

[![](localization-images/language-compare-sml.png "독일어 및 일본어 문자열 길이")](localization-images/language-compare.png#lightbox)

있음을 **설정을** 영어에서 (8 자)를 독일어로 번역 있지만 일본어로 2 자에 대 한 13 문자가 필요 합니다.

여기서는 표시 레이블 및 입력된 필드는 side-by-side-레이아웃은 레이블 길이 크게 달라질 수 있습니다 사용 하기가 어렵습니다. 종종 필드 위에 레이블을 표시 되는 위치는 레이아웃 화면의 전체 너비 레이블 및 입력을 모두 사용할 수 있으므로 지역화 하기가 쉽습니다.

일반적으로 작성 하는 경우 고정된 레이아웃 (특히 side-by-side-요소)에 영어 문자열 보다 자세한 너비 레이블 및 텍스트에 필요한 최소 50% 허용 합니다. 이 모든 문제를 해결 하지 않습니다 하지만 대부분의 경우에서 작동 하는 버퍼를 제공 합니다.

### <a name="input-validation"></a>입력된 유효성 검사

유효성 검사 규칙을 작성할 때 가정에 주의 하십시오. 유효한 단일 문자에 거의 모든 의미가 있으므로 "require" 영어로 세 개 이상의 문자를 입력 하는 텍스트 필드를 요구 하도록 보일 수 있습니다. 하지만 중국어 및 일본어로 단일 문자에 유효한 입력을 수 있습니다 및 유효성 검사를 메시지 "3 자 이상이 필요 합니다" 이해 되지 않는 해당 언어에 대 한 합니다.

웹 사이트 URL 문자를 사용 하 여 더 복잡 해질 ASCII 하위 집합으로 제한 되지 않습니다. 또는 다른 간단해 보이는 작업 같은 전자 메일 주소 유효성 검사.

대화 상자를 염두 국제화를 사용 하 여 유효성 검사 규칙 덜 제한적인 규칙을 선택 하거나 언어별로 다르게 작동할 수 있도록 논리를 작성을 작성 합니다.

### <a name="images-and-color"></a>이미지 및 색

모든 이미지 변경 해야 사용자의 언어 선택에 따라 합니다. 많은 아이콘이 나 사진 모든 사용자에 게 적합을 말하는 언어는 중요 하지 됩니다.
그러나 지역화에 적합 한 일부 리소스와 같은:

 - 이미지를 보여 주는 사람 또는 특정 위치 즉, 앱 전공 학생이 사용자에 게는 관련성이 로컬 사용자/위치 표시 된 경우.
 - 아이콘 –의 일부도 해 문화권별 수 있으며 앱을 로컬 이해를 반영 하는 이미지를 지역화 하 여는 데 쉽게 만들 수 있습니다.
 - 색 – 일부 문화권 이해 색 다르게 – red 하나의 지역이 긴 하지만 다른 행운을 빕니다 경고 의미할 수도 있습니다. 색을 지역화 하는 메커니즘을 구축 해야 하는지 여부를 확인 하려면 앱을 디자인할 때 모국어를 사용 하 여 확인 합니다.


### <a name="videos-and-sound"></a>비디오 및 소리

비디오 및 사운드 있는 특수 한 과제 비교적 쉽게 변환 하는 문자열을 얻을 수 있지만, 여러 voiceover 기록을 추적 하기 때문에 응용 프로그램을 지역화 하는 경우 또는 비디오 클립 비용이 많이 들고 어렵습니다 둘 다를 수 있습니다.

비디오 및 사운드 파일의 여러 복사본 (특히 경우 사용자는 많은 언어에 지역화 또는 많은 미디어 파일이) 응용 프로그램의 크기를 증가할 크게 수 있습니다. 사용자가 앱을 설치한 후이 또한로 인해 사용자 환경이 저하 속도가 느린 네트워크에서 필요한 언어 자산만 다운로드 하는 것이 좋습니다.

여러 가지 방법으로 지역화 문제를 해결 하는 경우가 종종 있습니다 – 가장 중요 한 점은 사전으로 간주 하 여 맡기십시오를 설계 된 응용 프로그램을 확인 합니다.


### <a name="dates-times-numbers-and-currency"></a>날짜, 시간, 숫자 및 통화

.NET 서식 지정 함수를 사용 하는 경우를 소수 구분 기호는 올바르게 구문 분석 (변환 예외 throw를 방지) 문화권을 지정 해야 합니다. 예를 들어 1.99와 1,99은 로캘에 따라 유효한 10 진수 표현입니다.

데이터가 알려진된 소스에서 제공 되는 경우 (ie. 사용자 고유의 코드 또는 사용자가 제어 하는 웹 서비스에서) 표준 영어 언어 서식에 InvariantCulture 같은 서식 지정과 일치 하는 문화권 식별자를 하드 코딩할 수 있습니다.

```csharp
double.Parse("1,999.99", CultureInfo.InvariantCulture);
```

데이터를 입력 하 여 앱 사용자가 되는 경우 해당 로캘을 나타내는 CultureInfo 인스턴스를 사용 하 여 구문 분석 합니다.

```csharp
double.Parse("1 999,99", CultureInfo.CreateSpecificCulture("fr-FR"));
```

참조 된 [숫자 문자열 구문 분석](https://msdn.microsoft.com/library/xbtzcc4w(v=vs.110).aspx) 하 고 [구문 분석 하는 날짜 및 시간 문자열](https://msdn.microsoft.com/library/2h3syy57(v=vs.110).aspx) 추가 정보에 대 한 MSDN 문서.

### <a name="right-to-left-rtl-languages"></a>오른쪽에서 왼쪽 (RTL) 언어

예를 들어, 아랍어, 히브리어 및 우르두어와 같은 일부 언어에서는 오른쪽에서 왼쪽으로 읽혀집니다.
이러한 언어를 지 원하는 응용 프로그램 오른쪽에서 왼쪽 판독기에 대 한 예를 들어 조정 하는 화면 디자인을 사용 해야 합니다.

 - 텍스트는 오른쪽 맞춤 이어야 합니다.
 - 레이블이 입력된 필드의 오른쪽에 표시 됩니다.
 - 기본 단추 배치 일반적으로 반전 됩니다.
 - 계층적 탐색 살짝 애니메이션 (및 기타 탐색 메타포 및 애니메이션) 방향을 사용 하는 컨텍스트를 되돌릴 수도 해야 합니다.

IOS 및 Android에는 오른쪽에서 왼쪽 레이아웃 및 위의 조정 하는 데 도움이 되는 기본 제공 기능을 사용 하 여 글꼴 렌더링을 지원 합니다. Xamarin.Forms는 RTL 렌더링을 자동으로 현재 지원 하지 않습니다.

### <a name="sorting"></a>정렬

동일한 문자 집합을 사용 하는 경우에 해당 알파벳의 정렬 순서를 다르게 정의 하는 다른 언어입니다.

참조를 [문자열 비교 세부 정보](https://msdn.microsoft.com/library/dd465121(v=vs.110).aspx#the_details_of_string_comparison) 에 [.NET Framework에서 문자열 사용에 대 한 모범 사례](https://msdn.microsoft.com/library/dd465121(v=vs.110).aspx) 언어 (CultureInfo) 정렬 순서를 적용 하는 위치 예입니다.

그럴 가능성은 모바일 플랫폼에 기본 제공 데이터베이스 기능 언어별 정렬 순서 비즈니스 논리에서 추가 코드를 구현 하는 데 필요한 수 있도록 지원 합니다.

### <a name="text-search"></a>텍스트 검색

작성 하 고 여러 언어를 사용 하 여 검색 알고리즘 염두에서 테스트를 확인 합니다. 고려해 야 할 사항은 다음과 같습니다.

 - 자동 완성-자동 완성 함수를 만든 경우 사용자의 언어와 관련 된 제안 원본 것을 확인 합니다.
 - 데이터를 일치 하는 쿼리는 특정에서 입력 쿼리를 검색 하는 언어는 해당 언어로 작성 된 콘텐츠를만 나 앱의 모든 콘텐츠에 대해 실행할 수 있습니까?
 - 형태소 분석 – 유사한 단어, 단어 루트 및 다른 검색 최적화를 검색할 검색어를 작성 하면 이러한 최적화를 지 원하는 모든 언어에 대 한 기본 제공?
 - – 정렬 (위 참조) 결과 제대로 정렬 되었는지 확인 합니다.


### <a name="data-from-external-sources"></a>외부 원본에서 데이터

Twitter에서 외부 원본의 데이터를 다운로드 하는 대부분의 응용 프로그램 및 날씨, 뉴스 또는 주식 시세를 RSS 피드입니다. 사용자에 게이 표시할 때 관련이 없는 또는 읽을 수 없는 정보 화면에 표시 됩니다는 가능성을 고려해 야 합니다.

사용자에 게 유용한 데이터를 표시 하는 앱을 확인 하 여 몇 가지 전략 가지가 있습니다.

 - 다른 원본-응용 프로그램 사용자의 언어 또는 로캘에 따라 서로 다른 원본에서 데이터를 다운로드할 수 있습니다. 로캘 뉴스, 날씨 및 주식 가격 북아메리카 피드에서 다운로드 하는 것 보다 더 적절을 해야 합니다.
 - 지역화 된 표시 – Twitter 또는 사진 피드를 표시 하는 경우 있습니다 해야 메타 데이터를 표시 (예: 소요 된 시간)가 자신의 언어에서 원래 언어로 콘텐츠 자체를 유지 하는 경우에 합니다.
 - 번역 – 들어오는 데이터의 기계 번역을 위해 앱으로 변환 옵션을 빌드할 수 있습니다. 자동 또는 사용자의 판단에 따라 –를 반드시이 이루어지는, 기계 번역 완벽 하지 되므로 하는 경우 사용자에 게 알릴 수 있습니다이!

이 오디오 트랙에 대 한 외부 링크에도 영향을 줄 수 있습니다 또는 변환 비디오 원본을 지정 하기 위해 미리 계획 해야 응용 프로그램을 디자인 하는 경우 콘텐츠는 사용자가 적절 하 게 바탕으로 하는 사용자 인터페이스에 콘텐츠가 표시 되지 것입니다 때 되도록 해당 언어입니다.


### <a name="dont-over-translate"></a>과도 하 게 변환 안 함

앱의 일부 문자열을 변환할 필요 하거나 최소한 변환기가 특별 한 주의가 필요 하지 않을 수 있습니다. 예제 포함 될 수 있습니다.

 - Url – URL을 나열 하는 경우 그렇지 언어에 따라 조정 될 필요가 없을 수 있습니다. 예를 들어 facebook.com이 자동으로 검색 기본 사이트에서 언어 번역이 필요 하지 않습니다. 로캘 관련 내용이 다른 사이트 및 yahoo.fr 나 yahoo.it yahoo.com 같은 다른 URL을 제공 하는 것이 좋습니다.
 - 전화 번호 – 다른 국가 코드를 사용 하 여 특히 또는 호출자는 특정 언어에 대 한 숫자입니다.
 - 연락처 세부 정보 – 주소 및 기타 정보는 언어 또는 로캘 달라질 수 있습니다.
 - 상표 및 제품 이름 – 일부 문자열 항상 동일한 언어로 작성 하기 때문에 변환 하는 기능이 필요 하지 않습니다.

마지막으로, 특정 문자열 특수 처리가 필요할 경우 변환기에 대 한 자세한 지침을 포함 해야 합니다.


### <a name="formatted-text"></a>서식 있는 텍스트

모바일 앱을 사용 하 여 문제가 하지 되므로 문자열 일반적으로 다양 하 게 서식이 지정 되지 않습니다. 그러나 서식 있는 텍스트 (예: 굵게 또는 기울임꼴 서식을) 앱에서 필요한 경우 확인 변환기 방법과 목적을 아는 형식을 입력으로 문자열 파일 저장이 올바르게 형식이 제대로 사용자에 게 표시 하기 전에 (ie. 실수로 말고 자체 형식 지정 코드를 사용자에 게 표시 됩니다).



## <a name="translation-tips"></a>변환 팁

응용 프로그램에서 사용 하는 문자열을 번역은 지역화 프로세스의 일부로 간주 됩니다. 일반적으로이 작업 번역 서비스를 아웃소싱 및 응용 프로그램 또는 비즈니스 모를 수 있는 다국어 담당자가 수행 됩니다.

다음 팁을 쉽게 정확 하 게 변환 하 고 지역화 된 앱의 품질을 향상 시킬 수 있는 문자열을 생성 하는 데 도움이 됩니다.



### <a name="localize-complete-strings-not-words"></a>전체 문자열에서 단어를 지역화 합니다.

경우에 따라 개발자 응용 프로그램에 걸쳐 이러한 다시 사용할 수 있도록 단일 단어 또는 문장 '조각'를 지정 하는 동안는 방법을 사용 합니다. 예를 들어 텍스트 "해야 메시지는 5 개입니다." 번역에 대 한 다음 문자열을 지정할 수 있습니다.

**잘못 된**:

```csharp
"You have"
"no"
"message"
"messages"
```

및 문자열 연결을 사용 하 여 코드에 올바른 구를에 즉석에서를 만들려고 시도 합니다.

**잘못 된**:

```csharp
"You have" + " " + numMsgs + " " + "messages"
"You have" + " no " + "messages"
```

**이 권장 되지 않습니다** 모든 언어에 대해 작동 하지 않는 하므로 짧은 각 세그먼트의 컨텍스트를 이해 하는 변환기에 대 한 어렵게 됩니다. 또한 문제가 발생할 수 있습니다 나중에 각기 다른 컨텍스트에서 사용 됩니다 (및 업데이트 한 후) 하는 경우 번역 된 문자열을 다시 사용을 수행 합니다.


### <a name="allow-for-parameter-re-ordering"></a>매개 변수 다시 정렬 하는 것에 대 한 허용

하지만 일부 프로그래밍 언어.NET 이미 개념을 지원 번호가 매겨진된 자리 표시자를 따라서 문자열에서 매개 변수의 순서를 지정 하는 추가 구문은 필요

**좋은**:

```csharp
"a {0} b {1} cde {3}"
```

일 수 있습니다 (여기서 자리 표시자의 순서와 위치 변경 됨) 다음 변환

```csharp
"{2} {3} f g h {0}"
```

및 토큰을 위한 변환기 같은 순서 여야 합니다. 자세한 내용은 각 자리 표시자 변환기에 대 한 문자열을 보낼 때 포함 해야 합니다.


### <a name="use-multiple-strings-for-cardinality"></a>카디널리티에 대 한 여러 문자열을 사용 합니다.

과 같은 문자열 방지 `"You have {0} message/s."` 각 상태에 대 한 특정 문자열을 사용 하 여 더 나은 사용자 환경을 제공 합니다.

**좋은**:

```csharp
"You have no messages."
"You have 1 messages."
"You have 2 messages."
"You have {0} messages."
```

표시 되는 숫자를 평가 하려면 앱에서 코드를 작성 하 고 적절 한 문자열을 선택 해야 합니다. 일부 플랫폼 (iOS, Android 등)의 현재 언어/로캘 설정을 기준으로 가장 좋은 복수 문자열을 자동으로 선택 하도록 기본 제공 기능이 있습니다.


### <a name="allowing-for-gender"></a>성별에 대 한 허용

라틴 문자 기반 언어는 종종 성별 주체에 따라 서로 다른 단어를 사용합니다. 성별에 대 한 앱 알고,이 반영 하기 위해 번역 된 문자열을 허용 해야 합니다.

영어, 특정 사용자 또는 앱의 사용자에 게 문자열이 참조 하는 위치에 보다 명확 하 게 대/소문자도 됩니다. 메시지와 같은 일부 사이트를 표시 하는 예를 들어 `"Bob commented on his post"` 남성, 여성, 및 이진이 아닌 또는 알 수 없는 성별에 대 한 문자열을 해야 합니다.

**좋은**:

```csharp
"{0} commented on his post"
"{0} commented on her post"
"{0} commented on their post"
```

### <a name="dont-reuse-strings"></a>문자열을 다시 사용 하지 마세요

또는 보다 정확 하 게 다시 사용 하지 마세요 문자열만 유사 하기 때문에 문자열 자체에 다른 용도로 또는 의미 합니다.

예를 들어: 앱을 설정/해제 스위치에 스위치 컨트롤에 대 한 텍스트 'on' 및 'off' 지역화 해야 한다고 가정 합니다. 수 값을 표시 합니다 해당 설정의 다른 곳에서 텍스트 레이블의 앱에서. 예를 들어 다른 문자열 (기본 언어 문자열 같은 경우에) 스위치의 상태와 전환 표시 하기 위해 사용 해야 합니다.

-   "On" – 스위치 자체가 표시
-   "Off" – 스위치 자체가 표시
-   "On" – 레이블에 표시
-   "Off" – 레이블에 표시

변환기에 대 한 최대 유연성을 제공 합니다.

-   디자인 상의 이유로 아마도 스위치 자체가 "on" 및 "off" 소문자를 사용 하지만 다음 표시 레이블 "On" 및 "Off" 대문자를 사용 합니다.
-   일부 언어 (번역 된) 단어 자동 완성 레이블을에 나타날 수 있지만 사용자 인터페이스 컨트롤에 맞게 약식 스위치 값을 해야 합니다.
-   또는 일부 언어의 문화적 경험에 대 한 "I"와 "O"를 사용 하지만 "On" 또는 "꺼짐" 레이블을 않으려고 할 수 있습니다 스위치의 렌더링 될 수 있습니다.

### <a name="translation-services"></a>번역 서비스

#### <a name="machine-translation"></a>기계 번역

앱에 변환 기능을 작성 하는 것이 좋습니다 합니다 [Azure Translator Text API](https://azure.microsoft.com/services/cognitive-services/translator-text-api/)합니다.

테스트 목적으로 개발 하는 동안 앱에서 일부 지역화 된 텍스트를 포함 하도록 다양 한 온라인 번역 도구 중 하나로 사용할 수 있습니다.

- [Bing 번역기](https://www.bing.com/translator/)
- [Google Translate](http://translate.google.com/)

도 많은 키워드가 사용할 수 있습니다. 기계 번역 품질 일반적으로 간주 되지 않는 응용 프로그램을 릴리스 수 있을 정도의 검토 및 전문 변환기 또는 모국어 테스트 먼저 없이 합니다.

#### <a name="professional-translation"></a>전문 번역

문자열에 수행 되며 자체 변환기를 해당 하는 요금에 대 한 완성 된 번역을 제공 하 게 배포 하는 전문 번역 서비스도 있습니다.

가장 유명 서비스 중 하나인 [LionBridge](http://www.lionbridge.com/)합니다. 대부분의 전문 서비스는 모든 일반적인 파일 형식 문자열, XML, RESX 및 POT/PO 등을 지원 합니다.
